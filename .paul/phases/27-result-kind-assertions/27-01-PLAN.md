## Goal
Add a strict Python-side `ptc.expect_kind(value, kind)` helper so model-written code can assert result kinds explicitly and fail with clear wrong-kind errors instead of ad hoc dictionary checks.

## Purpose
Milestone 11 pivots from broader handle expansion to safer Python ergonomics. Phase 27 should land the smallest useful kind-assertion primitive in the live runtime, proving that `code_execution` can validate structured result kinds without drifting into a general validation framework or changing existing tool result shapes.

## Output
- A bounded runtime helper: `ptc.expect_kind(value, kind)`
- Clear wrong-kind / missing-kind failure behavior inside the Python runtime
- Focused execution-level proof in a dedicated test file
</objective>

## AC-1: Matching kinds pass through unchanged
```gherkin
Given a structured Python value that includes a `kind` field matching the caller's expectation
When `ptc.expect_kind(value, kind)` is invoked
Then it returns the original value unchanged so follow-on code can keep using that same structured object inline
```

## AC-2: Wrong or missing kinds fail clearly
```gherkin
Given a structured Python value whose `kind` field is missing or does not match the expected kind
When `ptc.expect_kind(value, kind)` is invoked
Then execution fails with a clear bounded error that names the expected kind and the actual kind state
```

## AC-3: The helper is proven in the real code_execution runtime without broadening scope
```gherkin
Given the live CodeExecutor runtime and deterministic local test fixtures
When focused tests exercise `ptc.expect_kind` success and failure paths
Then the helper behavior is proven end-to-end without introducing general shape validation, new handle classes, or tool-introspection behavior
```

  <name>Task 1: Implement a bounded runtime kind assertion helper</name>
  <files>src/python-runtime/runtime.py</files>
  <action>
    Add `ptc.expect_kind(value, kind)` to the Python runtime helper surface.
    Implementation requirements:
    - inspect only the top-level `kind` field on structured values
    - return the original value unchanged when the expected kind matches
    - raise a clear `ValueError` when `kind` is missing, non-string, or mismatched
    - include expected vs actual kind information in the failure message
    - keep the helper generic and small; do not introduce a broader validation DSL or type-specific schema framework
    - preserve existing helper behavior (`ptc.extract_handles`, `ptc.first_handle`, `ptc.read_text`, etc.) unchanged
    Avoid: recursive shape validation, new normalized result envelopes, handle-category expansion, or edits to TypeScript-facing docs in this phase.
  </action>
