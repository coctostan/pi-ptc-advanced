<objective>
## Goal
Add `ptc.run_tests(pattern)` as a first-class Python helper that runs Node's built-in `node --test` runner from the active Python execution workspace and returns a structured Phase 50 `ptc_report` with pass/fail metrics, duration, command metadata, warnings, and a bounded failures table.

## Purpose
Milestone 18 is about making `code_execution` produce small, well-shaped, debuggable answers with less agent friction. Agents currently write bespoke Python subprocess code and ad hoc TAP/string parsing to run focused tests from inside `code_execution`; this phase provides one ergonomic helper for the Node test-runner case while preserving sandbox policy boundaries.

## Output
- Runtime helper: `ptc.run_tests(pattern)` in `src/python-runtime/runtime.py`.
- Focused TDD proof for passing tests, failing tests, invalid input, and runner-unavailable behavior.
- README, generated `code_execution` guidance, and CHANGELOG updates documenting the helper and its Node-only/sandbox-limited scope.
</objective>

```gherkin
Then the helper passes the pattern directly to `node --test <pattern>` with no Python glob expansion and returns a `ptc_report` with metrics for total, passed, failed, skipped, duration, exit code, command, pattern, runtime cwd marker, and runner availability
```

```gherkin
Given the selected Node test pattern contains failing tests

```


<boundaries>
## DO NOT CHANGE
- Do not add npm or Python dependencies.
- Do not implement vitest, jest, pytest, package-script dispatch, watch mode, or cross-runner abstraction in this phase.
- Do not change Docker images, sandbox creation policy, `PTC_USE_DOCKER`, or `PTC_ALLOW_UNSANDBOXED_SUBPROCESS` semantics.
- Do not bypass the active Python sandbox by adding host-side arbitrary command execution or broad RPC command frames.
- Do not change `ptc.report(...)` shape, completed-result rendering, or existing helper return contracts.
- Do not edit installed Pi home-directory/runtime files; source changes stay in this repo.

## SCOPE LIMITS
- `ptc.run_tests(pattern)` is a convenience/reporting helper, not a replacement for PALS verification gates; APPLY/UNIFY still run explicit `npm test`, `npm run build`, and audit commands.
- The helper may report runner unavailability when `node` is absent in the active runtime; installing Node into Docker or selecting a different Docker image is deferred.
- Output parsing should be robust and bounded, but it does not need to preserve full TAP output or every diagnostic line.
- The public Phase 53 API is exactly `ptc.run_tests(pattern: str)`; timeout/options objects and alternate runner selection are deferred.
- Reports should avoid absolute host cwd values; use `cwd: "."` or runtime-workspace wording plus pattern/command metadata.
</boundaries>
