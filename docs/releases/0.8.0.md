# Release Notes ŌĆö 0.8.0

`0.8.0` is the first coherent personal-fork release baseline for `pi-ptc-next` after the release/package surface was aligned.

## What this release is for

This release packages the current working fork state into a versioned, documented baseline that is easier to maintain and verify locally.

The core value is unchanged:

- `code_execution` should invoke the same active Pi tool implementations the user sees in chat
- hashline-native `read` / `grep` / `edit` behavior should stay trustworthy inside Python workflows
- personal maintenance should be reproducible without relying on old session artifacts

## Install / update target

Install directly from the fork:

```bash
pi install git:github.com/coctostan/pi-ptc-next
```

If you already have the fork checked out locally, update your branch using your normal manual git workflow, then re-run the verification commands below.

## Verification commands

For the `0.8.0` baseline, the expected repo-local verification flow is:
```bash
npm run verify:personal
npm run verify:personal:full
npm run verify:ci
npm run verify:release-package
```
What they mean:
- `npm run verify:personal` ŌĆö focused verification for routine local maintenance
- `npm run verify:personal:full` ŌĆö higher-confidence full-suite verification
- `npm run verify:ci` ŌĆö GitHub Actions parity verification, matching the repo-owned workflow command used by `.github/workflows/ci.yml`
- `npm run verify:release-package` ŌĆö package metadata + `npm pack --dry-run` tarball verification for the current release surface

## Major shipped capabilities in this baseline

### Runtime / tool behavior
- `code_execution` routes overridden builtin tools through the active Pi executors visible in chat
- structured `details.ptcValue` payloads are preserved across the RPC boundary
- Python helper contracts and generated wrappers expose structured anchored result models for builtin tooling
- callable-tool policy behavior remains explicit, including `ptc.callable`, `ptc.policy`, and allowlisted-but-unavailable warnings

### Personal fork workflow
- repo-local maintenance commands are now stable and documented
- the preferred personal analysis profile is captured in `scripts/start-pi-ptc-full-tools.sh`
- `.github/workflows/ci.yml` now runs the repo-owned verification-only CI path via `npm run verify:ci`
- the old upstream PR-prep material is retained only as archived reference, not as the active operating path
### Release/package baseline
- package/project version markers now align at `0.8.0`
- the package surface includes a checked-in MIT license file
- package metadata now points at the personal fork repository surface
- release-package verification is now a first-class repo-local command

## Known limitations / deferred items

This release is intentionally not the end of release hardening.

Still deferred:
- no automated tagging or publish flow
- no automated git/PR workflow built into the repo
- no bridge teardown yet; that still depends on Pi exposing `getToolExecutor()`
Current CI state:
- verification-only workflow present at `.github/workflows/ci.yml`
- local parity command is `npm run verify:ci`
- release publication remains manual even though verification is now automated

Current dependency audit baseline:
- dependency audit: `0 critical / 2 high / 1 moderate / 1 low`
- this release documents that baseline but does not remediate it yet
