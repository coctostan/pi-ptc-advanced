# Release Notes — 0.16.0

`0.16.0` is the Milestone 17 compatibility-proof and release-readiness baseline. It records the audited Pi compatibility delta, aligns the runtime to the latest Mario-scope Pi packages, and integrates `code_execution` prompt metadata — without claiming a hard `@earendil-works/*` migration and without claiming a clean dependency audit.

## What this release is for

This release closes the Milestone 17 audit-and-alignment loop with a coherent release-candidate surface:

- Pi compatibility evidence is captured as a durable audit artifact
- the runtime aligns with the latest Mario-scope Pi packages used by the active extension API
- `code_execution` is surfaced through Pi prompt metadata in the default `Available tools` section
- the dependency-audit baseline is explicitly acknowledged through `.paul/dean-baseline.json`

The fork lineage is unchanged:

- original upstream: [`cegersdoerfer/pi-ptc`](https://github.com/cegersdoerfer/pi-ptc)
- ongoing fork repository: [`coctostan/pi-ptc-next`](https://github.com/coctostan/pi-ptc-next)
- current package publish target: `pi-ptc-advanced`

## Install / update target

Install directly from the fork repository:

```bash
pi install git:github.com/coctostan/pi-ptc-next
```

The package metadata inside the repo now targets `pi-ptc-advanced@0.16.0`. As with prior baselines, this repo documents the install/update path but does not automate npm publish or git workflow operations.

## Verification commands

For the `0.16.0` baseline, the expected repo-local verification flow is:

```bash
npm run verify:personal
npm run verify:personal:full
npm run verify:ci
npm run verify:release-package
```

What they mean:
- `npm run verify:personal` — focused verification for routine local maintenance
- `npm run verify:personal:full` — higher-confidence full-suite verification
- `npm run verify:ci` — GitHub Actions parity verification, matching the repo-owned workflow command used by `.github/workflows/ci.yml`
- `npm run verify:release-package` — package metadata + `npm pack --dry-run` tarball verification plus a clean temporary install proof for the current publish surface

The new focused regression test `test/release-readiness.test.ts` keeps release metadata, verification-script assertions, README/runbook baselines, the changelog `0.16.0` section, and this release note aligned. It is included in the standard test runs above.

## Major shipped capabilities in this baseline

### Pi compatibility audit (Phase 45)
- A 271-line compatibility-and-prompt-integration audit lives at `.paul/phases/45-pi-api-and-documentation-delta-audit/45-01-PI-COMPAT-AUDIT.md`.
- Top-level findings include the build-time peer-scope drift between Mario-scope and `@earendil-works/*` Pi packages, the lack of `code_execution` prompt metadata in earlier baselines, and the absence of an observability signal when the hashline executor bridge emits no executors.
- This release does **not** perform a hard migration to `@earendil-works/*` runtime packages.

### Runtime compatibility alignment (Phase 46)
- Package metadata, lockfile, and the active extension API target the latest **Mario-scope Pi packages** (`@mariozechner/*@0.73.1`).
- `sourceInfo` is preserved separately from PTC's internal `source` taxonomy so the active Pi context contract continues to work.
- The hashline executor bridge now emits a no-executor observability warning while preserving the builtin fallback.

### Prompt metadata integration (Phase 47)
- `code_execution` now registers Pi `promptSnippet` and active-only `promptGuidelines` so the default `Available tools` section can describe when Python-backed batching, aggregation, and compact-results work is preferred.
- Prompt-time auto-routing skips duplicate routing text when Pi `systemPromptOptions` already include equivalent `code_execution` guidance.
- PTC-managed custom tools preserve user-authored `promptSnippet` and `promptGuidelines` when registering with Pi.

### Release-readiness proof (Phase 48)
- Package metadata, `package-lock.json` root metadata, and `scripts/verify-release-package.sh` target `pi-ptc-advanced@0.16.0` consistently.
- `test/release-readiness.test.ts` adds focused drift coverage for the release baseline, the verification script, README/runbook baselines, the changelog `0.16.0` section, and this release note.
- README, CHANGELOG, and `docs/personal-fork-maintenance.md` all reference `pi-ptc-advanced@0.16.0` as the active baseline and link to this release note.

## Dependency-audit caveat

This release **does not claim a clean dependency audit**. The current acknowledged baseline at release-candidate time is recorded by DEAN:

- `npm audit --json`: **4 critical / 0 high / 3 moderate / 0 low**
- recorded in `.paul/dean-baseline.json` and valid through `2026-06-11`
- the user explicitly selected override during `/paul:plan` for Phase 48

If you intend to actually publish this baseline, treat the audit findings as a separate manual decision rather than an implicit clean-audit signal.

## Known limitations / deferred items

This release continues the manual publish model. Still deferred:

- automated tagging, npm publish, or release GitHub Actions automation
- automated git/PR workflow built into the repo beyond GitHub Flow planning gates
- bridge teardown; that still depends on Pi exposing `getToolExecutor()`
- the manual operator decision to publish after the documented verification gate passes
- dependency-advisory remediation for the acknowledged DEAN baseline

Current CI state:
- verification-only workflow present at `.github/workflows/ci.yml`
- local parity command is `npm run verify:ci`
- release publication remains manual even though verification is automated
